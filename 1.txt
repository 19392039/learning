代码中最可疑的直接原因
最高嫌疑是 Vector Part 4 的 TQue 数据类型不一致。
在 [chunk_bwd_dqkwg_vector.h (line 832)](C:/Users/qs/Documents/Codex/2026-07-24/https-github-com-flashserve-flash-linear/work/flash-linear-attention-npu-26.6.0/fla/ops/ascendc/gdn/chunk_gdn_bwd/chunk_bwd_dqkwg/op_kernel/chunk_bwd_dqkwg_vector.h:832) 中：
pipe->InitBuffer(outQue2, BUFFER_NUM, gSize * sizeof(float));
后面按照 float 分配、入队：
auto tensorDgOut = outQue2.AllocTensor<float>();   // line 913
...
outQue2.EnQue(tensorDgOut);                        // line 978
但出队时却使用 GType：
auto tensorDgOut = outQue2.DeQue<GType>();         // line 983
case 3 中：
GType = half
也就是同一个队列：
AllocTensor<float>
EnQue<float>
DeQue<half>
这属于明显的类型合同不一致。中间还把 float buffer 原地 cast 到：
tensorDgOut.ReinterpretCast<GType>()
这可能导致：
TQue 对 buffer 长度/element count 的解释不一致；
出队 LocalTensor 的地址范围或长度错误；
后续 DataCopyPad 从错误的 LocalTensor 视图写入 gmDg；
AIV 首先出现 MTE/GM 异常；
AIC 因等待 AIV 的 CrossCore flag 而同时出现超时类异常。
手写 JSON 使用的是 g=fp32，此时 GType=float，上述问题被掩盖：
AllocTensor<float>
DeQue<float>
这非常符合“手写 fp32 gate JSON 能过，同精度 fp16 gate 用例触发设备异常”的现象。
其他高风险代码
1. Workspace 被多个阶段复用
同一文件中：
gmMm6.SetGlobalBuffer(workspace + wsMm5Offset);
gmMm7.SetGlobalBuffer(workspace + wsMm5Offset);
也就是说 mm5/mm6/mm7 复用同一段 workspace。
needReduce == 0 时又有：
gmMul1.SetGlobalBuffer((__gm__ DataType *)ptrDq);
即把最终输出 dq 当成 Part 2 的临时 workspace。
这些别名只有在所有 SyncAll 和 AIC/AIV CrossCore flag 完全正确时才安全。case 3 会启动16个 block，相比手写用例的2个 block，更容易暴露阶段不同步或某个 AIV 提前异常的问题。
2. 另一个 TQue 释放类型不一致
Part 3 中：
auto tensorMul1InFp16 = inQue2.DeQue<DataType>();
auto tensorMul1InFp32 =
    tensorMul1InFp16.ReinterpretCast<float>();

inQue2.FreeTensor(tensorMul1InFp32);  // line 710
分配和出队类型是 DataType，释放的却是 reinterpret 后的 float view。这里同样可能破坏队列/UB buffer 的管理状态。
3. 唯一明确的 fixed-mode 标量 GM 读取
日志明确提到 scalar GM access。fixed 模式中最值得检查的是：
float dgLast = gmDgLast.GetValue(dgLastOffset);
位于同一文件第1200行。
对 case 3，理论上：
dgLastOffset = 0..63
dgLast workspace = 64 × sizeof(float) = 256 bytes
地址计算本身没有越界。因此如果 PC 最终映射到这条指令，更可能说明：
workspace 基址不正确；
tiling data 与 kernel 二进制版本不一致；
前面的 TQue/同步异常破坏了执行状态；
host 和 kernel 使用了不同版本的 tiling struct。
4. Host 完全没有参数保护
[aclnn_chunk_bwd_dqkwg.cpp (line 76)](C:/Users/qs/Documents/Codex/2026-07-24/https-github-com-flashserve-flash-linear/work/flash-linear-attention-npu-26.6.0/fla/ops/ascendc/gdn/chunk_gdn_bwd/chunk_bwd_dqkwg/op_host/op_api/aclnn_chunk_bwd_dqkwg.cpp:76) 中：
CheckFormat() -> ACLNN_SUCCESS
CheckShape()  -> ACLNN_SUCCESS
CheckDtype()  -> ACLNN_SUCCESS
因此即使输入、输出或已安装 OPP 的 dtype/shape 合同不一致，host 也不会拒绝，而是直接把指针交给 kernel，最终表现就是507015。
可以排除的原因
我按 case 3 计算了所有主要张量和 workspace 边界：
numChunks       = 16
blockDim        = 16
dgLast          = 256 bytes
mm5/mm6/mm7区   = 1,048,576 bytes
dsTemp          = 524,288 bytes
总user workspace = 1,573,120 bytes
q/k/v/do/dv、h/dh、dq/dk/dw 的最后一个 chunk、最后一个 head 的结束地址都没有超过对应分配大小。
因此没有证据表明这是 generator 产生了普通 shape 越界。更像是：
GType != float 路径中的 TQue 类型错误；
AIV 异常后引发 AIC 跨核等待异常；
workspace/output 高度复用使问题在多 block 下被放大。
仅凭未符号化的 current PC 不能把错误百分之百映射到某一行，但从代码审查和 case 差异看，ProcessPart4 的 EnQue<float> → DeQue<half> 是最直接、最有针对性的开发缺陷。
